# DH — платформа бегового сообщества

> «DH» — кодовое имя репозитория/проекта. Продукт-facing бренд —
> **DАЙ ХАРD**, беговое сообщество Чебоксар (хэштег `#diehardcheb`, тег-лайн
> «🏃 Воскресные длительные тренировки.»). Фронтенд использует монохромную
> палитру (чёрный → белый через серый) и временный текстовый логотип «DX»
> вместо отдельного лого-файла (прав на старый лого сейчас нет).

## О проекте

Веб-платформа для бегового сообщества. Сообщество организует events (события),
внутри которых есть несколько под-событий — разные локации/дистанции/темповые
группы (см. пример: `DX #294` — событие с двумя локациями "Город Мастеров" и
"Тачанка", а внутри каждой — несколько групп с разным темпом: `X-33 группа #1`,
`D-25 группа #2` и т.д.). Бегуны записываются на группы, тренируются, и по
каждому событию фиксируется список пробежавших. Результат (дистанция, темп,
время) либо загружается файлом (GPX/FIT), либо вводится вручную. По истории
участия строится рейтинг активности бегунов сообщества.

Старт — веб-приложение (адаптивное), API проектируется так, чтобы позже можно
было добавить нативное мобильное приложение поверх того же бэкенда.

## Домен (черновая модель, будет уточняться)

### Структура события

Два уровня вложенности: **Event → Group**. Отдельной сущности "локация" нет —
локация это просто поле группы.

- **Event** — событие (например `DX #294`): заголовок, дата, описание.
  Создаётся вручную с нуля каждый раз (без шаблонов/повторов на старте).
  Нет статуса "отменено/перенесено" и истории таких изменений — событие
  либо будущее, либо прошедшее (определяется по дате), поля редактируются
  напрямую.
- **Group** — группа внутри события (например `X-33 группа #1, темп
  5:40-5:30`): ссылка на событие, локация (текстовое поле), название/код
  дистанции, диапазон темпа, `target_distance_km` и `start_time` (эталонные
  дистанция и время старта — используются для автопроверки результатов, см.
  раздел «Результаты»). Бегун записывается и бежит именно в рамках группы.

### Участие: запись (намерение) vs факт пробежки

Это два разных, слабо связанных процесса:

1. **Signup** (запись/намерение) — бегун заранее отмечается на группу
   (аналог опроса на скриншоте). Это не гарантирует, что он реально бежал.
2. **AttendanceRecord** (факт участия) — источник истины о том, кто реально
   пробежал. Заполняется постфактум через CSV-импорт (см. раздел про CSV).
   Каждая запись изначально — просто имя/строка из CSV, без привязки к
   аккаунту. При импорте сразу получает `finish_status = finished`
   («пробежал») — сам факт попадания в CSV-список уже означает участие.
   Этот статус далее может быть автоматически пересчитан при загрузке
   `Result` (см. ниже).
3. Сопоставление — **автоматическое** в момент импорта, запись без аккаунта
   не остаётся (см. «Гостевые аккаунты и заявки» ниже): если `raw_email`
   точно совпадает с email зарегистрированного (не гостевого) аккаунта —
   `AttendanceRecord` сразу привязывается к нему; иначе на лету создаётся
   гостевой аккаунт и запись привязывается к нему.

### Профиль пользователя (Runner)

- **Обязательные поля**: имя, фамилия, email, пароль.
- **Необязательные поля**: город, пол, дата рождения, телефон, аватар.
- Пользователь может редактировать свои данные и менять пароль.
- **Роль** (`runner`/`organizer`/`admin`) назначается только вручную через
  SQLAdmin — самостоятельного запроса/повышения роли нет.
- **Регистрация**: email/пароль + подтверждение email (письмо со ссылкой
  активации). Требует SMTP (Яндекс.Почта, пароль приложения).
- **Восстановление доступа**: сброс пароля по email (стандартный флоу со
  ссылкой/токеном).
- **Публичный профиль** — виден всем участникам сообщества: имя, аватар,
  рейтинг и история участий (список пройденных `AttendanceRecord`/`Result`).
  Приватных данных (email, телефон, дата рождения, город, пол) в публичном
  профиле нет — только сам пользователь и Admin.

### Персональные данные (152-ФЗ)

- **Политика обработки персональных данных** — публичная страница
  `/privacy-policy` (без логина). Юридические реквизиты оператора и
  контактный email в тексте — заглушки, обязательные к заполнению реальными
  данными до продакшн-запуска (см. выделенный блок в начале страницы).
- **Согласие при регистрации** — обязательный чекбокс со ссылкой на
  политику; `accept_privacy_policy` валидируется и на бэкенде (`RegisterRequest`),
  не только на фронте. Момент согласия пишется в `User.privacy_accepted_at`.
- **Право на выгрузку данных** — `GET /users/me/export`, кнопка «Скачать мои
  данные» в профиле (вкладка «Безопасность») — отдаёт профиль, историю
  участий и записи на группы в JSON.
- **Право на удаление** — `DELETE /users/me` (требует пароль), кнопка
  «Удалить аккаунт» там же. Аккаунт удаляется безвозвратно; исторические
  `AttendanceRecord` не удаляются (чтобы не портить чужой протокол), а
  обезличиваются — `raw_email`/`raw_phone` очищаются, `runner_id` становится
  `NULL` (тот же паттерн, что при гостевом слиянии). Организаторам с
  созданными событиями самостоятельное удаление недоступно (`events.created_by`
  — `ON DELETE RESTRICT`) — нужно сначала передать события Admin'у.

### Гостевые аккаунты и заявки на объединение

Чтобы протокол и рейтинг не оставались пустыми до ручной модерации, у CSV-
записи всегда есть аккаунт сразу после импорта:

- **Guest-аккаунт** — обычная запись в `User` (`is_guest = true`), но с
  синтетическими email/паролем — залогиниться под ней нельзя. Имя/фамилия
  берутся из `raw_name` (разбивается по первому пробелу). Участвует в
  протоколе и рейтинге как обычный аккаунт.
- **Сопоставление CSV-строки с аккаунтом при импорте** — по приоритету:
  1. `raw_email` точно совпал с email зарегистрированного (не гостевого)
     аккаунта → эта запись сразу на него.
  2. Иначе — по имени/фамилии (без учёта регистра) ищется уже существующий
     гостевой профиль. Если он уже объединён с реальным аккаунтом
     (`merged_into_id` задан) → запись уходит прямиком на этот реальный
     аккаунт, без новой заявки. Если найден необъединённый гость — запись
     привязывается к нему повторно (не плодит дублей одного и того же
     человека при импорте разных событий).
  3. Иначе — создаётся новый гостевой аккаунт.
  Email-заглушки в файле не нужны: дедупликация гостей держится на
  имени/фамилии, а не на почте.
- **GuestClaim** — заявка «это я»: реальный зарегистрированный пользователь
  находит свой гостевой профиль (поиск по имени) и заявляет на него права.
  Статусы: `pending` / `approved` / `rejected`.
- **Модерация** — на уровне заявки, а не на уровне CSV-импорта: Admin
  подтверждает заявку в `/admin-tools/claims` (или инициирует объединение
  сам, без заявки, через `/admin-tools/guests`, — доступны оба пути).
  При подтверждении все `AttendanceRecord`/`Signup` гостя переносятся на
  реальный аккаунт (`merge_guest_into`); гостевой профиль **не удаляется**,
  а помечается `merged_into_id = <id реального аккаунта>` — остаётся в базе
  для истории/аудита, но дальше пустует (все его записи уже перевешаны).
  Остальные незакрытые заявки на тот же гостевой профиль автоматически
  отклоняются.

### Рейтинг

- **Rating** — агрегированная статистика активности бегуна: количество
  `AttendanceRecord` со статусом `finished`, привязанных к его аккаунту (за
  всё время и/или за период). Записи со статусом `DNF` в рейтинг не входят.

### Результаты (Result) и протокол группы

Группа отображается как протокол соревнования: список участников,
отсортированный по времени (лучшее — выше), с DNF отдельно/внизу. Плюс
третья группа — **«в пути»** (`pending`): записи со статусом `finished`, у
которых ещё нет `approved`-результата (только что импортированные/только
что созданный гостевой аккаунт, результат ещё не загружен). Без неё участник
после CSV-импорта не виден в протоколе вообще — это и есть весь смысл
группы: результат может подгрузиться позже, но сам факт участия виден сразу.

- `Result` связан 1:1 с `AttendanceRecord` — один результат на одно
  фактическое участие. Повторная загрузка перезаписывает предыдущий
  результат. Доступен для заполнения только после того, как
  `AttendanceRecord` привязана к аккаунту.
- Источник данных — либо файл (GPX/FIT), либо ручной ввод. Файл в
  приоритете: ручной ввод разрешён только если файла нет.
- Обязательный минимум полей: дистанция и время (темп и место в протоколе
  считаются автоматически). Из файла дополнительно сохраняем максимум того,
  что удаётся извлечь: трек (точки маршрута — для отображения на карте, как
  у `Group.route_gpx`), профиль/набор высоты, телеметрию из FIT, если она
  есть (пульс, каденс и т.п.).
- **`finish_status` (пробежал / DNF)** — пересчитывается автоматически при
  добавлении `Result`, по той же дистанции и допуску, что и в автопроверке
  ниже: если пройденная дистанция ≥ `target_distance_km` минус допуск (10%)
  → `finished`, иначе → `DNF`. До загрузки `Result` статус остаётся тем, что
  был проставлен при CSV-импорте (`finished` по умолчанию).
- **Статус модерации** (доверие к данным результата, отдельно от
  finish_status): `pending` (ждёт проверки Admin) / `approved` (данные
  подтверждены). В протокол по времени и в рейтинг попадают только
  `approved` + `finished` записи.
- **Автопроверка при загрузке файла** — сравнивает данные файла с
  эталонными `Group.target_distance_km` и `Group.start_time`:
  - дистанция в пределах ±10% от `target_distance_km`;
  - время старта в пределах ±1 часа от `Group.start_time`.
  - Оба условия выполнены → статус модерации сразу `approved`.
  - Хотя бы одно не выполнено → `pending`, уходит в очередь модерации Admin
    (как и любой ручной ввод).
- Ручной ввод — всегда `pending`, требует подтверждения Admin.

### Медиа

- **Event.cover_image** — обложка события, один файл на событие. Загружает
  Organizer или Admin.
- **EventPhoto** — отдельная таблица фотографий, привязанных к событию
  (many-to-one к `Event`). Заполняется постфактум (например, после
  тренировки), загружают Organizer/Admin. Публикуется сразу, без модерации.
- **Group.route_gpx** — GPX-файл планового маршрута группы, загружает
  Organizer при создании/редактировании группы (не обязателен для создания
  группы как такового — можно донести позже). Это отдельный файл от
  результатов бегунов: он описывает маршрут заранее, а не фактическую
  пробежку.
  - На фронте: интерактивная карта маршрута (трек поверх тайлов
    OpenStreetMap) + профиль высоты + кнопка «скачать GPX».
  - Бэкенд парсит GPX и отдаёт фронту точки трека/профиль высоты как JSON
    (переиспользует тот же GPX-парсер, что и для результатов бегунов).

### Роли

- **Admin** — полный доступ, включая CSV-импорт и модерацию заявок на
  объединение гостевых профилей (`/admin-tools` + SQLAdmin) — для **всех**
  событий, в том числе созданных организаторами.
- **Organizer** — создаёт и редактирует свои события/группы, грузит обложку,
  фото и GPX-маршрут группы. CSV-импорт и модерацию заявок не выполняет —
  это всегда делает Admin.
- **Runner** — обычный участник: запись на группу, загрузка/ввод своих
  результатов, просмотр рейтинга.

### Лимиты на загружаемые файлы (дефолты, можно поменять позже)
- Изображения (обложка события, фото) — до 10 МБ, форматы jpg/png/webp.
- GPX/FIT (маршрут группы, результат бегуна) — до 5 МБ.

## Стек

### Backend
- **Python 3.12+**, **FastAPI** (async, чистый API-бэкенд, без серверного
  рендеринга).
- **SQLAlchemy 2.0** (async) + **Alembic** для миграций.
- **Pydantic v2** для схем запросов/ответов.
- **PostgreSQL** как основная БД.
- Auth: email/пароль, JWT (access + refresh).
- Email (подтверждение регистрации, сброс пароля) — отправка через SMTP
  Яндекс.Почты (пароль приложения), без отдельного email-провайдера на
  старте.
- Парсинг GPX/FIT — синхронно в момент загрузки файла (без очереди на
  старте). Если нагрузка вырастет — вынести в фоновую задачу
  (Celery/RQ) как отдельное решение, не блокирует текущую архитектуру.
- Хранение файлов — локально на VPS (volume в контейнере). Если понадобится
  масштабирование — миграция на S3-совместимое хранилище (MinIO) как
  отдельная задача.

### Frontend
- **React + Vite + React Router**, TypeScript.
- Отдельное SPA, общается с backend только через REST/JSON — так же будет
  работать и будущий мобильный клиент.
- **Leaflet + OpenStreetMap** (бесплатные тайлы) для отображения GPX-трека
  маршрута группы, плюс простой график профиля высоты.

### Админка

Два раздела с разной аудиторией:

- **`/admin-tools`** (Jinja2, серверный рендеринг, свой стиль, не похож на
  SQLAdmin) — основной интерфейс для Organizer и Admin на каждый день:
  события/группы (создание, редактирование, обложка, фото, GPX-маршрут) —
  доступно обеим ролям, Organizer видит и правит только свои события, Admin —
  все; импорт CSV, заявки на объединение гостевых профилей, гостевые
  профили без заявки, подтверждение результатов на проверке — только Admin.
  - **Вход из фронтенда без повторного логина**: кнопка «Admin Tools» в шапке
    сайта видна только `organizer`/`admin`, ведёт на
    `GET /admin-tools/sso?token=<access_token>` — бэкенд проверяет JWT
    (тот же, что уже выдан фронту при логине) и заводит ту же
    cookie-сессию, что использует `/admin-tools`. Если токен истёк — откат на
    обычную форму логина `/admin-tools/login` (email/пароль).
  - Сессия — тот же `SessionMiddleware`/cookie, что и у SQLAdmin, поэтому
    админ, зашедший через SSO в `/admin-tools`, уже залогинен и в SQLAdmin
    тоже (кнопка «SQLAdmin →» в шапке `/admin-tools`, видна только Admin).
    Organizer в SQLAdmin по-прежнему не пускают (роль проверяется отдельно).
- **SQLAdmin**, встроена в FastAPI-приложение (тот же процесс, та же БД).
  Полный CRUD по всем таблицам, в первую очередь — создание пользователей и
  назначение ролей (см. «Роли» выше — это единственный способ стать
  Organizer/Admin). Используется редко, для операций, которые сознательно не
  вынесены в `/admin-tools`. Доступ — только для роли admin.

- **mypy** — типы.

### Инфраструктура
- **Dev**: Docker Compose (`docker-compose.yml`) — `api` (uvicorn --reload),
  `db` (postgres), `frontend` (vite dev server), всё в контейнерах.
- **Prod**: сервер слабый, поэтому в Docker (`docker-compose.prod.yml`)
  остаются только `api` и `db`. Фронтенд собирается статикой (`vite build`,
  предпочтительно не на самом сервере) и отдаётся **нативным nginx на
  хосте** (`nginx/dh.conf`, ставится в `/etc/nginx/sites-available`, не
  контейнер) — он же терминирует TLS (certbot) и проксирует `/api`,
  `/admin`, `/admin-tools` на `api` через `127.0.0.1:8000`, отдаёт `/media`
  напрямую с диска. Подробности и порядок деплоя — `DEPLOY.md`.
- Секреты прод-окружения — в `.env` в корне репозитория (гитигнорится,
  шаблон в `.env.example`), читается и `docker compose` для подстановки
  переменных, и передаётся в контейнер `api` как `env_file`. Приложение
  отказывается стартовать при `ENVIRONMENT=production`, если
  `SECRET_KEY`/`ADMIN_SECRET_KEY`/`INITIAL_ADMIN_PASSWORD` остались
  дефолтными или короче 32 символов.
- Один Git-репозиторий (монорепо): `backend/`, `frontend/`,
  `docker-compose.yml`/`docker-compose.prod.yml` в корне.

## Рабочий процесс с бегунами через CSV (текущее решение)

Формат файла: разделитель колонок — точка с запятой (`;`). Колонки:
`first_name`, `last_name` (обязательны, раздельно — чтобы не гадать, в каком
порядке в файле написаны имя и фамилия), `email`, `phone`, `result` (не
обязательны).
`result` — просто флаг, не хранимое значение: `0` → статус `DNF` (бежал, но
не финишировал), любое другое непустое значение или отсутствие колонки →
`finished` (поведение по умолчанию не меняется для старых файлов без этой
колонки).

1. Admin загружает CSV со списком пробежавших — в том числе для событий,
   созданных организаторами. Каждая строка сразу получает аккаунт: по email
   находится зарегистрированный пользователь, либо на лету создаётся
   гостевой аккаунт (см. «Гостевые аккаунты и заявки» выше) — записи без
   аккаунта не остаётся, протокол не пустует.
2. Бегун регистрируется в системе самостоятельно (email/пароль).
3. Если бегун понимает, что его результат попал в систему как гостевой
   профиль (email в CSV не совпал с тем, каким он зарегистрировался, или он
   зарегистрировался позже) — он сам находит этот профиль по имени и подаёт
   заявку «это я»; Admin подтверждает, и все результаты гостя переезжают на
   реальный аккаунт.
4. Дальше бегун может сам загрузить результат (GPX/FIT) или ввести его
   вручную по своим прошедшим записям.

## Конвенции

- Код и идентификаторы — на английском, комментарии/докстринги — только
  когда нужно объяснить неочевидное решение.
- Все схемы БД — через Alembic-миграции, без ручных правок схемы.
- Тесты — pytest, покрывать в первую очередь бизнес-логику (парсинг файлов,
  расчёт рейтинга, привязку CSV-записей к аккаунтам).
